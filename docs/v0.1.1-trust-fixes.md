# v0.1.1 Trust Fixes

This release is about trust, not more rules.

The Dify Compose scan exposed the biggest product risk in v0.1: the scanner can find useful issues, but the report overstates confidence by turning a large upstream Compose file into `0/100` without enough context.

## Goal

Make reports distinguish:

- active risks: likely apply to the current default stack
- conditional risks: apply only when an optional Compose profile or optional service is enabled
- template risks: found in `.env.example`, example files, or default placeholders
- hygiene notes: useful hardening advice that should not dominate the score

The user should leave with a prioritized fix list, not just fear.

## Product Rules

### 1. Compose profiles

Parse `profiles` on services.

Default behavior:

- Services with no `profiles` are active.
- Services with `profiles` are conditional.
- Conditional findings are shown, but do not affect the primary risk score by default.

CLI additions:

```bash
selfhosted-doctor scan ./stack --profile milvus
selfhosted-doctor scan ./stack --profile milvus --profile opengauss
selfhosted-doctor scan ./stack --all-profiles
```

Scoring behavior:

- default: score active findings only
- `--profile`: score default services plus selected profiles
- `--all-profiles`: score everything

Report behavior:

- Terminal report should include a `Conditional` section when profile-gated findings exist.
- JSON should expose enough metadata for AI/MCP to explain why a finding is conditional.

Acceptance test:

- A service under `profiles: [milvus]` that publishes a public port should not lower the default scan score.
- The same service should lower the score when `--profile milvus` or `--all-profiles` is used.

### 2. Template and example env files

Classify env files by path:

- `.env` = active
- `.env.local`, `.env.production`, `.env.prod` = active
- `.env.example`, `.env.sample`, `.env.template`, files under `examples/` = template

Template secret behavior:

- Do not emit high-severity `plaintext-secret` findings for template files.
- Emit info or low findings titled like `Default secret in template env file`.
- Recommendation should say: copy this file to `.env`, then rotate/change every default before deploying.

Real env behavior:

- Keep high severity for literal secrets in active env files.

Acceptance test:

- `.env.example` with `DB_PASSWORD=difyai123456` does not produce a high finding.
- `.env` with `DB_PASSWORD=difyai123456` still produces a high finding.

### 3. Compose default secrets

Current parser treats `${VAR:-secret}` as a reference and skips it. That misses an important self-hosted risk.

Add detection for Compose fallback defaults:

```yaml
POSTGRES_PASSWORD: ${DB_PASSWORD:-difyai123456}
```

Behavior:

- If the service is active, emit a high finding: `Default secret fallback in service environment`.
- If the service is conditional, emit a conditional finding.
- Redact the fallback value.

Acceptance test:

- `${DB_PASSWORD:-difyai123456}` is flagged.
- `${DB_PASSWORD}` is not flagged.
- Secret values are still never leaked in JSON, Markdown, terminal, MCP, or AI explain output.

### 4. Better scoring

Replace pure linear per-finding scoring with capped risk buckets.

Suggested scoring model:

```text
score = 100 - activePenalty

activePenalty =
  publicDataServiceExposure capped at 40
  privilegedOrHostControl capped at 30
  activePlaintextSecrets capped at 25
  sensitiveAppWithoutAccess capped at 25
  publicAppExposure capped at 20
  reliabilityHygiene capped at 10
  imagePinning capped at 5
```

Important properties:

- One exposed database should hurt a lot.
- Ten unpinned images should not destroy the score.
- A large upstream Compose file should not automatically become `0/100`.
- Conditional/template findings should not affect the default score.

Acceptance test:

- A clean stack still scores >= 90.
- A stack with one public database scores clearly risky.
- A stack with 25 unpinned images but no active high risks stays above 80.
- Dify default scan should not be `0/100` due only to profile-gated services and `.env.example`.

### 5. Report copy

Change report language from "all findings are equal" to prioritized guidance.

Terminal shape:

```text
selfhosted-doctor report

Risk score: 72/100
Files scanned: 2
Active findings: 1 high, 9 medium, 12 low
Conditional findings: 4 high, 18 medium, 8 low
Template findings: 10 info

Fix first
- vastbase: Database port 5432 is public when profile vastbase is enabled
- opengauss: Runs privileged when profile opengauss is enabled

Active
...

Conditional
...

Template
...
```

Exact score above is illustrative, not required.

### 6. Documentation

Update README:

- Explain that the default score is for active/default services.
- Document `--profile` and `--all-profiles`.
- Add a short "Why is `.env.example` not high severity?" note.

Update launch kit:

- Do not claim the score is a universal security grade.
- Say it is a prioritized local configuration review.

## Non-goals

- Do not connect to Docker daemon in this release.
- Do not call Cloudflare API.
- Do not add auto-fix.
- Do not hide real high-risk findings to make the score look nicer.

## Claude Code Prompt

Implement `docs/v0.1.1-trust-fixes.md`.

Keep the existing architecture: deterministic rules produce findings, reporters render the shared report object, AI/MCP only consume the report.

Focus on:

1. service profile parsing and scan options
2. finding classification for active / conditional / template
3. `.env.example` secret severity downgrade
4. `${VAR:-default}` secret fallback detection
5. capped scoring
6. terminal/JSON/Markdown output updates
7. focused tests, including a Dify-like fixture

Do not add Docker daemon access, Cloudflare API calls, auto-fix, or a Web UI.
